The Experiment That Changed Everything

A full walkthrough of the anomaly detection experiment at Capgemini — from problem framing to the labeling bug discovery that 28% recall with zero model change.

Experimentation Shadow Mode A/B: Anomaly Detection System
1
Problem Framing
The rule-based system was failing silently
The existing system was flagging anomalies with 52% recall — meaning it missed nearly half of all real fraud events. Stakeholders wanted a new ML model, but the harder question wasn't how to build it. It was: how do we know if the new model is actually better in production, not just on a test set? Two metrics pulled in opposite directions: precision and recall. Stakeholders had different thresholds for acceptable false positives.
Baseline recall: 52%
2
Experiment Design
Three-phase shadow mode evaluation
Phase 1 — Established a 30-day baseline by logging every flag and outcome from the live rule-based system.

Phase 2 — Ran the new Random Forest + Gradient Boosting ensemble in shadow mode for 2 weeks — generating predictions alongside the live system without acting on them. Like-for-like comparison on identical data.

Phase 3 — Pre-registered primary metrics (recall, precision, F1) and guardrail metrics (false positive rate capped at baseline, analyst review volume capped at +10%) before looking at results. This prevents p-hacking.
3
The Discovery
The bug hiding in disaggregated analysis
During shadow mode review, I noticed 60% of false positives clustered in a single customer segment despite it representing only 18% of the dataset. Disaggregated fairness audit surfaced the disparity. Tracing false positives back to features revealed this segment had been systematically mislabeled at data ingestion — a pipeline bug from months earlier that went undetected because no one had checked segment-level error rates.

The insight: the model wasn't broken — the data feeding it was.
False positives in segment: 60% of all FPs · Segment size: 18% of data
4
Corrective Action
Four steps — upstream fix, not model tuning
Step 1: Halted downstream decisions relying on flags from the affected segment.
Step 2: Traced the labeling error to its root cause in the ingestion pipeline and fixed it with engineering.
Step 3: Relabeled affected records using the corrected logic and retrained on clean data.
Step 4: Formalized disaggregated segment-level audit as a mandatory pre-deployment check — added to the team's model review checklist.
